hashcode和equals

为什么要重写 hashcode 和 equals 方法？

重写hashcode作用是给确定每个对象在集合中位置（也可以用来快速判重，hash不同一定不相等，否则再进行equals判定，使用短路判断节约性能）。

重写equals作用是给确定两个对象在业务逻辑上是否相等。

在HashMap存入元素时，先通过key的hashcode计算元素储存的位置，如果存储位置上已经有元素了，就使用equals判断两个key是否相同，如果相同覆盖这个节点，如果不同则是发生hash值冲突及hash相等但是内容不相同，则在此节点后插入链表节点，同理继续插入新节点。

为什么重写equals()就一定要重写hashCode()方法？

两者事实关系如下：

  • 如果两个对象相同（即用equals比较返回true），那么它们的hashCode值一定要相同！！！！；
  • 如果两个对象不同（即用equals比较返回false），那么它们的hashCode值可能相同也可能不同；
  • 如果两个对象的hashCode相同（存在哈希冲突），那么它们可能相同也可能不同(即equals比较可能是false也可能是true)
  • 如果两个对象的hashCode不同，那么他们肯定不同(即用equals比较返回false)

所以从程序实现原理上来讲的话，既需要equals()方法，也需要hashCode()方法。那么既然重写了equals（），那么也要重写hashCode()方法，以保证两者之间的配合关系。

如果仅仅重写了equals（），而没有重写hashCode()方法，会出现什么情况？

字段属性值完全相同的两个对象因为hashCode不同，所以在hashmap中的table数组的下标不同，从而这两个对象就会同时存在于集合中（违背了set中元素不能重复的原则），所以重写equals()就一定要重写hashCode()方法。

如果不涉及到使用哈希表定位，那么可以不重写hashcode()，否则建议进行重写。
